# AGENTS.md

OpenCode harness: Rust DAG compiled to WASM and loaded in-process by the plugin (no separate runtime service).

## High-value commands

```sh
cargo test                    # fast unit tests for DagEngine
make wasm                     # rebuild plugin/wasm/ from src/lib.rs (requires wasm-pack)
cargo run -- install          # install embedded agents + skills into ~/.config/opencode/
cargo run -- install --force  # overwrite existing installed agent markdown and skills
openagent-harness install     # same install path via built binary
cargo fmt && cargo clippy     # default rustfmt/clippy config
```

- Release CI runs: `make wasm`, `cargo test`, `cargo clippy -- -D warnings`.
- If you change Rust WASM exports or `plugin/harness.ts`'s WASM interface, regenerate `plugin/wasm/` and commit the generated artifacts.

## Entrypoints that matter

- `src/lib.rs`: `DagEngine`, embedded `AGENTS`, wasm exports (`DagEngine` wrapper + `get_agent_configs()`)
- `plugin/harness.ts`: plugin runtime, session/event wiring, ACP calls, error classification, fallback decisions
- `src/main.rs`: `openagent-harness install [--force]` CLI
- `src/install.rs`: writes embedded agents to `~/.config/opencode/agents/` and skills to `~/.config/opencode/skills/`
- `src/agents.rs`: `parse_agent_frontmatter` — parses `model` and `fallback_models` from agent `.md` frontmatter; `AgentConfig` — the parsed result
- `src/skills.rs`: `SKILLS` constant with `include_str!` for each embedded skill; `install()` writes to `~/.config/opencode/skills/{name}/SKILL.md`
- `plugin/wasm/`: generated by `make wasm` **and committed** (`.js`, `.d.ts`, `.wasm`)

**Skills system:**
- Skills are stored at `~/.config/opencode/skills/` — shipped by this repo via `src/skills.rs`.
- `cargo run -- install` installs embedded skills alongside agents.
- Currently shipped: `caveman` (ultra-compressed output mode, ~65% token reduction).
- Agents declare `skills:` in frontmatter; they must load the skill via the skill tool before acting.
- **caveman** — loaded by: builder, debugger, planner, consultant, explorer, researcher, reviewer, vision, docs-writer. NOT loaded by: orchestrator, builder-junior.
- **git-workflow** — builder: manages worktree lifecycle for parallel junior workers.
- **git-worktree** — builder-junior: enforces atomic seed-commit + autosquash-fixup pattern inside worktree.
- **azure-workflow** — builder, debugger: hard read-only constraint for Azure operations.
- **pr-workflow** — orchestrator: PR creation and lifecycle management.

**Builder worktree model:**
- Builder applies `git-workflow` skill before spawning junior workers.
- Creates isolated `git worktree` per junior: path `../work-<task-id>`, branch `ai/*`.
- Each builder-junior spec must include: `worktree_path`, `branch_name`, `base_branch`, `card_number`.
- Builder-junior applies `git-worktree` skill (seed commit + autosquash fixups) inside its worktree.
- Worktrees are cleaned up by builder after juniors complete.

## Runtime behavior (don't mis-assume)

- Plugin loads WASM synchronously: `initSync(readFileSync(...))` before hooks run.
- Startup calls `get_agent_configs()` once; missing agent `.md` files are auto-installed on first boot.
- Startup also calls `agent_fallback_configs_json()` and passes the result to `dag.set_agent_fallbacks()` to register per-agent fallback chains.
- Tick loop (`setInterval`, 500ms) only starts tasks (`dag.tick()` + session create/send + `dag.task_started()`).
- Task completion/failure is event-driven via `session.idle` / `session.error` → `dag.process_event()`.
- ACP send endpoint is `POST /session/{id}/prompt_async` (fire-and-forget).
- Error classification happens in the TypeScript plugin (`classifyError` in `harness.ts`), not in WASM.
- Fallback decisions are made in the plugin; the WASM engine exposes `try_fallback()` to atomically advance the model and reset task state to Pending.
- Plugin errors are logged; they are not propagated to OpenCode.
- Orchestrator flow: planner may clarify via question tool → planner saves a plan artifact under `.opencode/plans` → user approval (via question tool) → orchestrator calls `submit_plan` with `native_dispatch: true` → native dispatch loop (`harness_dispatch_tasks` polls → spawn Task tool calls in parallel → `harness_task_complete` per task) → report results. The orchestrator never submits a workflow without user confirmation.

## Non-obvious gotchas

- Model string parsing: `provider/model`; if no `/`, provider defaults to `anthropic`; empty string means omit model.
- State persistence is in-memory only; restart loses workflow/task state.
- Rust 2024: `gen` is reserved; do not use it as an identifier.
- Fallback chain behavior: when a provider returns 429/5xx or a transient error, the task is re-queued with the next model in its chain. The task is only marked `Failed` after all fallbacks are exhausted or the error is classified as terminal (auth failure, content policy, etc.).
- `model_attempt` on a task tracks which model in the chain is active: `0` means the primary model, `1` means the first fallback, and so on. This field is visible in workflow snapshots.
- Agent fallback configs are loaded from frontmatter at startup via `agent_fallback_configs_json()` + `set_agent_fallbacks()`; they apply to every task dispatched to that agent unless the task supplies its own `fallback_models`.
- The orchestrator agent needs `permission: question: allow` in its frontmatter to use OpenCode's built-in question tool for plan approval. Without this, the question tool calls will be denied.
- The planner agent now also needs `permission: question: allow` so it can resolve missing requirements before saving a plan artifact.
- Native dispatch: when `submit_plan` is called with `native_dispatch: true`, the orchestrator must run the `harness_dispatch_tasks` / Task tool / `harness_task_complete` loop manually.
- External skills (git-workflow, azure-workflow, pr-workflow) are NOT shipped by this repo — they must be installed separately. If a declared skill is missing at `~/.config/opencode/skills/`, the agent that needs it will be blocked.
- Builder-junior operates inside a git worktree, not the main repo. Its working directory is `../work-<task-id>` on branch `ai/*`. Never run git operations in the main repo from a junior context.
- Caveman skill is auto-installed by `cargo run -- install` (unlike external skills). It compresses subagent prose output by ~65% via `src/skills.rs` → `~/.config/opencode/skills/caveman/SKILL.md`. Orchestrator and builder-junior intentionally excluded.
